## 13. Notable Structural Gaps

These are cases where collected data is particularly meaningful but finds no path into the AI.

### 13.1 `price_level` and `average_check_per_person`
~~Collected on the menu page and used to set price expectations in the UI. Neither field appears in any AI prompt. A budget café and a fine-dining restaurant receive identically structured prompts with no price-register guidance.~~

**Resolved (April 2026):** `price_level` is now injected in all three AI features as a content register label (budget / casual / mid-range / premium). `average_check_per_person` UI input was removed; AI injection skipped — §16.1 covers the register signal sufficiently.

### 13.2 `business_visual_identity`
~~Stores photography style, preferred angles, brand colours, and interior style. The photo guidance in generated suggestions comes from `recognizable_interior_identity` (a text field from brand profile) and `visual_character`. The richer structured data in `business_visual_identity` is never consumed.~~

**Partially resolved (April 2026):** `photography_style` and `photo_types_to_avoid` are now injected in `get-quick-suggestions` photo direction block. Interior style, brand colours, and preferred angles remain unused.

### 13.3 `business_audience_profile`
~~Stores detailed audience segments, competitor names, and market position — all AI-generated. The content creation AI only reads `target_audience` (a simpler text/JSONB field on `business_brand_profile`). Competitor data, audience channel preferences, and market position are entirely unused.~~

**Dropped April 2026 (migration `20260420000007`):** Table confirmed 0 rows and no code references. Skipped for AI injection (§16.12) — assessed as overlapping with `target_audience`; no distinct prompt slot identified. Table removed.

### 13.4 Menu item engagement history
~~`menu_items_normalized` tracks `total_times_posted`, `avg_engagement_rate`, and `last_posted_date` per dish. This is never read. Dish selection in suggestions and strategy uses only a 14-day recency check on `daily_suggestions.menu_item_name` — there is no performance-based selection.~~

**Closed (April 2026 cleanup):** All three engagement columns confirmed empty (write-back pipeline never built; §16.9 skipped). Columns dropped from `menu_items_normalized` in migration `20260419000002`. If social analytics integration ships in future, columns would need to be re-added before §16.9 can be implemented.

### 13.5 `is_seasonal` and `is_limited_time` flags
~~The AI dish selector in `get-weekly-strategy` Phase 2b actively chooses dishes based on seasonality (derived from the description text), but the explicit `is_seasonal` and `is_limited_time` flags on `menu_items_normalized` are not queried.~~

**Resolved (April 2026):** Both flags are now queried in Phase 2b dish selector. `is_limited_time` dishes are prioritised; `is_seasonal` dishes are boosted in the current month. `is_signature` (§16.4) also implemented in the same selector pass.

### 13.6 `business_goals`
~~The business goal structure (fill specific time slots, promote specific menu items, build for a season) is populated but not fed into any content creation AI. The weekly strategy makes goal decisions (goal_blend, goal_mode per slot) entirely from its own signals (weather, events, day of week) without knowing what goals the owner actually set.~~

**Dropped April 2026 (migration `20260420000007`):** Table confirmed 0 rows and no code references. Goals UI was test-only and never shipped. Table removed.

**Design note for future rebuild:** Business goals are fundamentally about one thing — more customers, more revenue. That translates not just to footfall posts but to the full content mix: a weekend-heavy bar focuses different post types than a lunch café targeting office workers. A goals model should capture *which levers matter for this specific business* (peak day, underperforming service period, new menu launch, seasonal push) and let those drive `goal_blend` and post-type distribution in the weekly strategy, rather than storing abstract objective records.

### 13.7 `specials`
~~Active promotions, happy hours, and seasonal events are tracked in the `specials` table but are never passed to any AI. The weekly strategy reads `contextual_calendar` (national holidays) but not business-specific special offers.~~

**Dropped April 2026 (migration `20260420000007`):** Table confirmed 0 rows and no code references. Specials UI was never shipped. Table removed. If business-specific promotion injection is revisited (§16.7), the table and UI would need to be rebuilt.

**Suggested injection point:** Brand block in all three AI features, alongside `business_character`. Map numeric value (1–4) to a human-readable label: _budget_, _casual_, _mid-range_, _premium/fine-dining_.
