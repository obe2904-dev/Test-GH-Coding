## Why Static Primary/Secondary Fails

### Failure 1: Time-of-Day Mismatch

**Example**: Tuesday 10:00 AM post

**Static Logic**:
- Primary archetype = casual_dining
- casual_dining says: post lunch content Tue-Fri 10:00-13:00
- **Result**: Lunch post at 10:00 AM

**Reality**:
- At 10:00 AM Tuesday, audience is in "coffee break" mode, not "lunch planning" mode
- Cafe Faust at 10:00 AM is functioning as coffee shop (morning coffee crowd)
- Better: Coffee/pastries post at 08:00 AM for 09:00-10:00 visits
- Lunch post should wait until 11:00-12:00 (closer to actual lunch decision time)

**The Problem**: Static archetype doesn't account for "what is the business right now?"

---

### Failure 2: Seasonal Business Shift

**Example**: May (outdoor season) vs November (indoor season)

**Static Logic**:
- Primary = casual_dining (50% menu posts)
- 50% of 4 weekly posts = 2 menu posts per week
- Programme rotation gives: 1 lunch dish, 1 dinner dish
- **Result**: Same content mix all year

**Reality May**:
- Outdoor season = cocktails on patio is 60% of weekend revenue
- Brunch on outdoor seating is growing segment
- Indoor lunch is slower (people want lighter, quicker meals in nice weather)
- **Should be**: 2 cocktail posts, 1 brunch post, 1 lunch post

**Reality November**:
- Indoor season = hearty lunch is 50% of weekday revenue  
- Evening comfort dining growing
- Outdoor cocktails irrelevant (patio closed)
- **Should be**: 2 lunch posts, 1 dinner post, 1 behind-the-scenes (cozy atmosphere)

**The Problem**: Static archetype can't adapt to seasonal revenue shifts

---

### Failure 3: Event-Driven Business Mode

**Example**: Week of December 15-21 (peak Christmas party season)

**Static Logic**:
- Primary = casual_dining
- Posts: 1 lunch, 1 dinner, 1 behind-the-scenes, 1 craving_visual
- **Result**: Normal weekly content

**Reality**:
- This week: 8 Christmas party group bookings (each 15-25 people)
- Revenue: 70% from Christmas parties, 30% from regular business
- Marketing opportunity: Drive more party bookings (each booking = €800-1200)
- **Should be**: 3 Christmas party posts (availability, package details, atmosphere), 1 regular post

**The Problem**: Static archetype ignores event-driven business mode shifts

---

### Failure 4: Weekly Rhythm Mismatch

**Example**: Friday afternoon post

**Static Logic**:
- Primary = casual_dining
- Friday = good day for dinner post (Wed-Sat dinner optimal)
- **Result**: Dinner dish post Friday 14:00

**Reality**:
- Friday 14:00: Audience is planning Friday evening + Saturday

- **Better**: Cocktails post Friday 14:00 (nightlife_bar archetype)
  - Drives Friday evening drinks + Saturday return visits
  - Higher revenue (cocktails + late-night sales)
  - Aligns with audience intent (weekend party planning)

**The Problem**: Static archetype picks wrong programme for temporal context

## Root Cause: Static Classification for Dynamic Business

**The Fundamental Error**: Treating business archetype as a fixed property when it's actually **temporal and contextual**

### What Cafe Faust Actually Is:

Not: "A casual dining restaurant with secondary brunch/nightlife"

But: "A **temporally dynamic hybrid** that operates as:
- Coffee shop (Mon-Fri mornings)
- Lunch restaurant (Mon-Fri midday)
- Cafe (Mon-Fri afternoons)
- Dinner restaurant (Mon-Sat evenings)
- Nightlife bar (Fri-Sat late-night)
- Brunch destination (Sat-Sun mornings)
- Family restaurant (via børnemenu, weekends)

**With seasonal emphasis shifts**:
- May-September: Outdoor/cocktails emphasis (40% revenue shift)
- October-April: Indoor/comfort food emphasis
- December: Event/celebration emphasis (Christmas parties)
- January: Health/light menu emphasis (New Year resolutions)

**And event-driven mode changes**:
- Mother's Day week: Brunch emphasis
- Christmas season: Party venue emphasis
- Valentine's Day: Romantic dining emphasis
- Summer festival weeks: Late-night bar emphasis"

---

## The Solution: Temporal Context-Aware Archetype Resolution

### Principle: Programme + Context → Archetype → Rules

Instead of:
```
Business → Static Archetype → Static Rules → Content
```

Use:
```
Programme + Week Context + Time Target → Dynamic Archetype → Context-Aware Rules → Content
```

### 3. Programme Priority Scoring (Context-Aware)

Instead of static recency/frequency/revenue scoring, add **temporal relevance multiplier**:

```typescript
function calculateProgrammePriority(programme, week_context) {
  // Base scoring (existing)
  const recency_score = calculateRecencyScore(programme)  // 0-50
  const frequency_score = calculateFrequencyScore(programme)  // 0-30
  const revenue_score = programme.revenue_weight || 0  // 0-20
  
  const base_score = recency_score + frequency_score + revenue_score  // 0-100
  
  // NEW: Temporal relevance multiplier
  const temporal_multiplier = calculateTemporalRelevance(
    programme.temporal_relevance,
    week_context
  )  // 0.5 to 2.0
  
  const final_score = base_score * temporal_multiplier
  
  return final_score
}

function calculateTemporalRelevance(temporal_relevance, week_context) {
  let multiplier = 1.0  // Default (neutral)
  
  // Month match
  if (temporal_relevance.peak_months.includes(week_context.month)) {
    multiplier += 0.3  // Boost by 30%
  }
  if (temporal_relevance.avoid_months.includes(week_context.month)) {
    multiplier -= 0.3  // Reduce by 30%
  }
  
  // Season match
  if (temporal_relevance.peak_seasons.includes(week_context.season)) {
    multiplier += 0.2
  }
  
  // Event match
  const matching_events = week_context.events.filter(event =>
    temporal_relevance.event_affinity.includes(event.name)
  )
  if (matching_events.length > 0) {
    multiplier += 0.5  // Strong boost for event match
  }
  
  // Weather boost (for outdoor-related programmes)
  if (temporal_relevance.peak_seasons.includes("outdoor_season") &&
      week_context.weather.outdoor_suitability === "good") {
    multiplier += 0.2
  }
  
  // Clamp multiplier to reasonable range
  return Math.max(0.5, Math.min(2.0, multiplier))
}
```

## How This Solves the Day/Week/Month/Season Problem

### Time of Day → Programme Selection + Archetype Rules

**Problem**: "At 10:00 AM Tuesday, is Cafe Faust a coffee shop or lunch restaurant?"

**Solution**: 
- Programme temporal_relevance specifies peak_times
- "Coffee & Pastries" programme: peak_times = ["07:00-11:00"]
- "Pariserbøf Lunch" programme: peak_times = ["11:00-15:00"]
- Temporal scoring at 10:00 AM:
  - Coffee programme gets slight boost (still in window)
  - Lunch programme gets neutral (not yet in window)
- But: Post timing != visit timing
- **Real logic**: "We're posting at 10:00 AM to drive 12:00-14:00 visits"
  - So lunch content at 10:00 AM post time is correct
  - peak_times refers to visit window, not post window

**Refinement Needed**: Separate `visit_window` from `posting_window`

function calculateProgrammePriorities(programmes, recent_posts, week_context) {
  programmes.forEach(programme => {

      recencyScore(programme) +

## Summary: From Static to Temporal

**Static Archetype Problem**:
- "Cafe Faust is casual dining (primary) with brunch/nightlife (secondary)"
- Doesn't account for: time of day, day of week, month, season, events
- Results in: cocktails on Sunday 9:00, lunch emphasis in May outdoor season, missed Mother's Day opportunity

**Temporal Solution**:
- Each programme has temporal_relevance (when it's most relevant)
- Week context analyzed (season, month, events, weather)
- Programme priority = base_score × temporal_multiplier
- Results in: cocktails emphasized in May outdoor season, brunch boosted for Mother's Day week, lunch naturally de-emphasized in nice weather

**Owner Burden**: Zero (AI generates temporal_relevance during onboarding, owner can review/adjust)

**Flexibility**: Infinite (handles any day/week/month/season pattern, any event, any weather shift)

**Predictability**: High (deterministic scoring with logged multipliers for debugging)

This transforms the system from "what type of business are you?" (static) to "what opportunities exist this week for each programme?" (dynamic).
