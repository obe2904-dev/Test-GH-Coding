# Generate Text From Idea: Data Map

This document maps the data that flows into `generate-text-from-idea/index.ts` and explains what comes from the idea, what comes from Brand Profile, and what other data shapes the generated text.

## Pipeline Overview

The flow is:

1. Resolve business context.
2. Resolve content context from the idea.
3. Select a CTA.
4. Build the prompt.
5. Call the model.
6. Post-process the text.

The prompt is assembled from two main inputs:

- The idea itself, which provides the subject and content angle.
- The business / Brand Profile data, which provides voice, identity, timing, and guardrails.

## Input Sequence Overview

This is not one bulk input. The generator builds the final prompt in a fixed sequence:

| Step | Input | AI task | Output |
|---|---|---|
| 1 | Business / Brand Profile data, business location, opening hours, language, paid/free tier | Resolve the business context and establish the venue frame | Normalized business context with brand voice, location, language, and timing facts |
| 2 | Idea fields such as `contentType`, `title`, `menuItemName`, `menuItemDescription`, `whyExplanation`, `photoIdea`, `weeklyPlanContext` | Resolve the content context and decide what the post should be about | Content context with the subject, angle, menu anchor, and weekly-plan details |
| 3 | `ctaIntent`, `goalMode`, `bookingLink`, `typicalClosings`, `reservationRequired`, `acceptsWalkIns`, `suggestionId` | Select the CTA and determine whether it should be strict or soft | Chosen CTA object and CTA mode |
| 4 | Business context, content context, CTA, prompt components, brand rules | Build the final prompt that will be sent to the model | One assembled prompt with system guidance, user instructions, and constraints |
| 5 | Assembled prompt, model settings, language mode | Generate the text from the prompt | Draft output text from the model |
| 6 | Draft output, brand validation rules, safety rules, formatting rules | Clean, validate, and finalize the generated text | Final post text ready for use |

In short: the AI does not get one undifferentiated blob. It receives layered context that is assembled step by step, then turned into the final prompt.

### Post-processing and validation

After generation, the system applies cleanup and safety passes:

- Voice validation against `brandProfileV5`.
- Targeted voice fixes for critical violations.
- Spell correction when needed.
- Removal of banned closers.
- Removal of AI-style dashes.
- Removal of meta-instruction leakage.
- Scene-format validation for behind-the-scenes and atmosphere posts.
- Hashtag generation from city, content type, menu item, and business name.

### Detailed Prompt Section Flow

| Step | Input | AI task | Output |
|---|---|---|---|
| System prompt | Brand Profile voice data, brand guardrails, content anchors, tier-specific rules | Set the voice boundary, tone, style, and forbidden patterns for the generation run | System instruction block that controls how the model should write |
| User prompt | Idea fields, menu fields, weekly-plan fields, content angle, visual cues | Turn the idea into a concrete writing brief with subject, angle, and factual content | User instruction block that tells the model what to write about |
| CTA | CTA intent, goal mode, closings, booking availability, walk-in support, suggestion context | Choose the call to action and decide whether it should be soft, strict, or omitted | Selected CTA text and CTA mode attached to the prompt |
| Context injection | Business context, opening hours, location, language, menu lookups, paid/free tier state | Add the factual frame so the output stays grounded in the correct venue and time | Enriched prompt context with venue, timing, and lookup data |
| Prompt assembly | System prompt, user prompt, CTA, context injection | Merge all prompt pieces into one structured prompt payload | Final prompt ready for the model call |
| Post-processing | Draft model output, validation rules, safety rules, formatting rules | Clean the draft, fix violations, and normalize the final text | Final publishable post text |

### Section Notes

- The system prompt is mostly the rule layer, not the content layer.
- The user prompt is the content layer: it carries the actual idea to be written into text.
- The CTA is selected separately so it can be controlled independently from the body copy.
- Post-processing happens after the model response and is not part of the creative prompt itself.

### AI Persona

The AI persona is built into the system prompt, not added as a separate step. It comes mainly from Brand Profile voice data such as `brandTone`, `brandWritingRules`, `brandGoodExamples`, `brandAvoidExamples`, `brandSignaturePhrases`, `voiceRationale`, `venueIdentity`, `businessCharacter`, `identityKeywords`, and `voiceConstraints`.

In practice, that means the persona is the voice layer that tells the model how to sound while the user prompt tells it what to write.

## Short Summary

The generator is driven by three layers:

1. The idea gives the topic, hook, and content angle.
2. The Brand Profile gives the voice, identity, and do-not-say guardrails.
3. Business, menu, and timing data provide the factual frame so the output stays grounded.

### Data Lifecycle

**1. Menu Discovery** → `menu_sources` created  
**2. Extraction Initiated** → `menu_results_v2` row with `status='queued'`  
**3. AI Processing** → Updates with `structured_data`, `service_periods`, `language_code`  
**4. Completion** → `status='done'`, trigger fires  
**5. Normalization** → Items flattened to `menu_items_normalized`  

**Re-extraction (Same URL):**
- New `menu_results_v2.id` created
- Same `source_id` preserved
- Old items in `menu_items_normalized` marked `is_active=false`
- New items inserted with `is_active=true`

**Menu Deletion:**
- All items for that `menu_url` marked `is_active=false`
- Data preserved for analytics, hidden from queries

## Classification Pipeline

### Stage 1: AI Extraction (menu-extract-v2)
**Location**: `supabase/functions/menu-extract-v2/index.ts`

The edge function processes menu URLs and extracts structured data using GPT-4o-mini.

#### AI Prompt Extracts:
- `menuTitle` - Menu name (e.g., "BRUNCH", "Cocktails", "AFTEN")
- `availabilityTime` - Time range if present (e.g., "09.00-14.00")
- `availabilityDays` - Days of week if specified
- `categories[]` - Menu sections with items

#### Example AI Output:
```json
{
  "menuTitle": "Cocktails",
  "availabilityTime": null,
  "categories": [
    {
      "name": "Cocktails",
      "items": ["Faust Stormy", "Amaretto Sour", ...]
    }
  ]
}
```

### Stage 2: Time Period Parsing (menuPeriodParser.ts)
**Location**: `supabase/functions/_shared/menuPeriodParser.ts`

Converts extracted data into structured menu periods with timing information.

#### Function: `parseMenuPeriods()`
Processes categories and assigns time periods based on:

1. **Menu-level availability** (`availabilityTime`)
2. **Category-level time ranges**
3. **Inferred from category names**
4. **Inferred from item types**

#### Menu Period Types:
```typescript
type MenuPeriodType = 
  | 'breakfast'   // 07:00-11:00
  | 'brunch'      // 09:00-14:00
  | 'lunch'       // 11:30-15:00
  | 'afternoon'   // 14:00-17:00
  | 'dinner'      // 17:00-22:00
  | 'late_night'  // 22:00-02:00
  | 'all_day'     // 00:00-23:59
  | 'other'       // 00:00-23:59 (unknown)
```

#### Time Inference Logic:
```typescript
if (startTime === '00:00' && endTime === '23:59') {
  type = 'all_day'
} else if (startHour < 11) {
  type = 'brunch'
} else if (startHour >= 17) {
  type = 'dinner'
} else {
  type = 'lunch'
}
```

### Stage 3: Service Period Detection (menu-extract-v2)
**Location**: `supabase/functions/menu-extract-v2/index.ts` (Lines ~1330-1365)

Converts parsed menu periods into canonical service period taxonomy.

#### Detection Priority Order:

**1. Title Pattern Matching (Highest Priority)**
```typescript
if (menuTitle.includes('bar') || menuTitle.includes('cocktail') || menuTitle.includes('drink')) {
  rawPeriodName = 'bar'
} else if (menuTitle.includes('brunch') || menuTitle.includes('morgenmad')) {
  rawPeriodName = 'brunch'
} else if (menuTitle.includes('frokost') || menuTitle.includes('lunch')) {
  rawPeriodName = 'frokost'
} else if (menuTitle.includes('aften') || menuTitle.includes('dinner')) {
  rawPeriodName = 'aften'
}
```

**2. All-Day Menu Detection**
```typescript
if (startTime === '00:00' && endTime === '23:59') {
  rawPeriodName = 'all_day'
} else if (type === 'all_day' || type === 'other') {
  rawPeriodName = 'all_day'
}
```

**3. Time-Based Inference (Fallback)**
```typescript
const startHour = parseInt(menuPeriods[0].startTime.split(':')[0])
if (startHour < 11) {
  rawPeriodName = 'brunch'
} else if (startHour >= 17) {
  rawPeriodName = 'dinner'
} else {
  rawPeriodName = 'lunch'
}
```

**4. Default**
```typescript
rawPeriodName = 'lunch'  // If no other detection works
```

### Stage 5: Database Normalization (Trigger)
**Location**: `supabase/migrations/20260507000001_create_menu_normalization_worker.sql`

When extraction completes (`status = 'done'`), items are flattened into `menu_items_normalized`.

#### Normalization Priority:

**1. Parent Menu Service Periods** (Most Reliable)
```sql
IF NEW.service_periods IS NOT NULL AND ARRAY_LENGTH(NEW.service_periods, 1) > 0 THEN
  v_service_periods := NEW.service_periods;
END IF;
```

**2. Menu Title Pattern Matching** (Fallback)
```sql
CASE 
  WHEN LOWER(v_menu_title) LIKE '%brunch%' THEN v_service_periods := ARRAY['brunch'];
  WHEN LOWER(v_menu_title) LIKE '%bar%' OR LOWER(v_menu_title) LIKE '%cocktail%' THEN v_service_periods := ARRAY['bar'];
  ...
END CASE;
```

**3. Empty Array** (Default)
```sql
ELSE v_service_periods := ARRAY[]::TEXT[];
```

### Key Implementation Details

#### 1. Service Period Matching (Quick Suggestions - Paid Tiers Only)

**File**: `supabase/functions/get-quick-suggestions/index.ts` (Lines 628-653)

**Tier Distinction:**

**Free Tier:**
- Uses `business_profile.key_offerings` directly (simple dish name list)
- No service period detection (no `business_programme_profiles`)
- No rotation queue (no `menu_items_normalized`)
- No rationales generated (just dish names)

**Paid Tiers (Smart/Pro):**
```typescript
if (isPaidTier) {
  // Detect current service period
  const servicePeriodResult = await detectServicePeriod(supabase, businessId, currentTimeHHMM)
  const currentServicePeriod = servicePeriodResult.currentPeriod  // 'brunch'|'lunch'|'dinner'|'all_day'

  // Get rotation queue filtered by service period
  const rotationQueue = await getMenuRotationQueue(supabase, {
    businessId,
    servicePeriod: currentServicePeriod,  // Only dishes for current service
    lookbackDays: 90
  })

  // Match dish by name + service_period
  const queueItem = rotationQueue.find(item => 
    item.menu_item_name === dishName &&
    (item.service_period === currentServicePeriod || currentServicePeriod === null)
  )

  // Fallback: name-only match (legacy compatibility)
  const fallbackItem = !queueItem && rotationQueue.find(item => 
    item.menu_item_name === dishName
  )

  const menuItemId = (queueItem || fallbackItem)?.menu_item_id
}
```

**Why this works**:
- Prevents wrong UUID when duplicates exist (e.g., lunch vs dinner variants)
- Service period filtering reduces ambiguity
- Fallback ensures compatibility with old data
- Free tier bypasses all menu classification systems

**Rotation Fairness:**
- Considers BOTH published posts (`status = 'published'`) AND scheduled posts (`status = 'scheduled'`)
- Uses `posted_at` for published content, `scheduled_for` for scheduled content
- Prevents double-booking the same dish (won't suggest a dish already scheduled)
- Sorts by "never posted → oldest first" to ensure fair rotation across entire menu

### Rotation Tracking Queries

#### Find all drink posts:
```sql
SELECT pp.*, min.category_type, min.item_name, min.service_periods
FROM published_posts pp
JOIN menu_items_normalized min ON pp.menu_item_id = min.id
WHERE min.category_type = 'drinks'
ORDER BY pp.posted_at DESC;
```

#### Food vs drink distribution:
```sql
SELECT 
  CASE 
    WHEN min.category_type = 'drinks' THEN 'Drinks'
    WHEN min.category_type = 'coffee' THEN 'Coffee'
    ELSE 'Food'
  END as post_category,
  COUNT(*) as total_posts,
  COUNT(DISTINCT min.id) as unique_dishes
FROM published_posts pp
JOIN menu_items_normalized min ON pp.menu_item_id = min.id
WHERE pp.menu_item_id IS NOT NULL
  AND pp.posted_at >= NOW() - INTERVAL '90 days'
GROUP BY post_category;
```

#### Most posted dishes (with service period):
```sql
SELECT 
  min.item_name,
  min.service_periods,
  min.category_type,
  COUNT(*) as post_count,
  MAX(pp.posted_at) as last_posted,
  AVG(EXTRACT(EPOCH FROM (NOW() - pp.posted_at)) / 86400) as avg_days_between
FROM published_posts pp
JOIN menu_items_normalized min ON pp.menu_item_id = min.id
WHERE pp.posted_at >= NOW() - INTERVAL '90 days'
GROUP BY min.id, min.item_name, min.service_periods, min.category_type
ORDER BY post_count DESC
LIMIT 20;
```

#### Check for duplicate menu items (same name, different UUID):
```sql
SELECT 
  item_name,
  service_periods,
  COUNT(*) as variant_count,
  ARRAY_AGG(id) as uuids
FROM menu_items_normalized
WHERE business_id = 'your-business-id'
  AND is_active = true
GROUP BY item_name, service_periods
HAVING COUNT(*) > 1;
```

### Category Type Classification

**File**: `supabase/functions/menu-sync/index.ts` (Lines 220-250)

Menu items are classified by `category_type`:

```typescript
function classifyCategoryType(categoryName: string): string {
  const lower = categoryName.toLowerCase()
  
  // Drinks (cocktails, wine, beer, spirits)
  if (lower.includes('cocktail') || lower.includes('apéritif') || 
      lower.includes('drinks') || lower.includes('vin') || 
      lower.includes('øl') || lower.includes('spiritus')) {
    return 'drinks'
  }
  
  // Coffee & Tea
  if (lower.includes('kaffe') || lower.includes('coffee') || 
      lower.includes('the') || lower.includes('tea')) {
    return 'coffee'
  }
  
  // Desserts
  if (lower.includes('dessert') || lower.includes('kage')) {
    return 'dessert'
  }
  
  // Kids menu
  if (lower.includes('børnemenu') || lower.includes('kids')) {
    return 'kids_menu'
  }
  
  // Sides
  if (lower.includes('tilbehør') || lower.includes('sides')) {
    return 'sides'
  }
  
  // Appetizers
  if (lower.includes('forretter') || lower.includes('starter')) {
    return 'appetizer'
  }
  
  // Default: main course
  return 'main'
}
```

**Stored in**: `menu_items_normalized.category_type`  
**Used for**: Rotation filtering, analytics, content type detection

## Common Issues & Fixes

### Issue 1: Cocktails Menu Assigned to Brunch
**Cause**: Time inference runs before keyword detection  
**Fix**: Check for bar/cocktail keywords BEFORE time inference

### Issue 2: All-Day Menu Assigned to Brunch
**Cause**: `00:00` start time passes `< 11` check  
**Fix**: Detect `00:00-23:59` as special case for `all_day`

### Issue 3: Orphaned Service Periods
**Cause**: Menu has service period without matching programme profile  
**Fix**: Create programme profile or update menu classification

### Issue 4: Old Items After Re-extraction
**Cause**: Trigger deleted by `menu_result_id` (new ID each time)  
**Fix**: Soft-delete by `menu_url` to handle same-URL updates
